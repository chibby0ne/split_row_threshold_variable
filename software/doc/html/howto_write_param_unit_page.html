How to write a new parameter unit

Before starting to implement or convert existing code to a new parameter unit, please read Simulation chain concept. The chapter explain the design idea behind the software.

For writing a new parameter unit for the simulation you have to do the following steps:

  • Create a parameter class for the parameter unit
  • Writing the parameter unit itself
  • Documentation of the Module
  • Make the Module Visible for the Configuration

In the following these steps are explained in more detail at the example of a standard-compliant parameter unit.

Create a parameter class for the parameter unit

Writing the parameter class for the parameter unit is pretty much the same as writing a parameter class for a module. Please read Parameter units before proceeding. The difference is that now output parameters are defined using the Param_Out container.

Adding parameters to the class

Parameters of the parameter unit are added to the public part of the class. Use Doxygen comments here (triple slash for short description). A detailed description is optional, but should be given, when applicable. All input parameters have to use the container class Param, all output parameters the container class Param_Out, respectively.

public:
    ....

    // Input Parameters

    Param<unsigned int> mcs_field;


    Param<unsigned int> duration;


    // Output Parameters

    Param_Out<unsigned int> num_info_bits;

    Param_Out<Mapper_Share::MAPPINGS_ENUM> mapping;

    ...

Initialization of parameters

Like for parameter classes of the modules, a default value has to be given for the input parameters. The output parameters do not need a default value, however, they have to be registered with a string: 

protected:

    virtual void Set_Default_Values()
    {
        // Input parameters
        mcs_field.Init(3, "mcs_field", param_list_);
        duration.Init(0, "duration", param_list_);

        // Output parameters
        num_info_bits.Register("num_info_bits", param_list_);
        mapping.Register("mapping", param_list_);
        ...

        mapping.Link_Value_String(Mapper_Share::MAP_QPSK_PI4, "MAP_QPSK_PI4");
        mapping.Link_Value_String(Mapper_Share::MAP_16_APSK,  "MAP_16_APSK");
        mapping.Link_Value_String(Mapper_Share::MAP_32_APSK,  "MAP_32_APSK");
    }

Document the Set_Default_Values() method with doxygen comments that state the default values for the module! Only then the user knows about the default values from the documentation!

Add Automatic Configuration Capability

The parameter unit is configured automatically by the environment. As for functional modules, writing a custom Config_From_Data_Struct() method is possible, though.

The parameter class for the parameter unit is complete now.

Writing the parameter unit itself

Writing a parameter unit is the same as writing a module, see How to write a new functional module. But instead of calculating data for the output port, the purpose of the Run() function is to write values to the output parameters.

int Param_Unit_GMR::Run()
{
    if(param_list_.config_modified())
        Init();

    return 0;
}

void Param_Unit_GMR::Init()
{
    Get_Num_Info_Bits();
    Get_Mapping();

    param_list_.config_modified(false);
}

Also, make sure that the parameter unit is derived from the Base_Interface class and its parameter class.

Documentation of the Module

Remember to document the parameters and the functionality of the parameter unit with doxygen comments such that other users can use your unit!

Make the Module Visible for the Configuration

The CSE environment allows for an automatic configuration via XML configuration files, see The < param_unit > tag. One important part of the configuration file are the parameter units, see Parameter Unit Concept.

To enable all these advanced features, you have to register the parameter unit in a unqiue list. The environment provides the template function Manage_Module_Config::Register_Module to do this. 

Manage_Module_Config::Registger_Module<Param_Unit_GMR>();

You have to register your parameter unit before reading a XML configuration file with the Manage_Module_Config class.
